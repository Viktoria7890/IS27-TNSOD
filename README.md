# Лабораторная работа 1: UserService

**Задача**: Написать доменный сервис для выполнения CRUD операций с пользователями с использованием gRPC.

## Шаги:
1. Создание модели User с полями: `id`, `login`, `password`, `name`, `surname`, `age`.
2. Реализация методов:
    - `CreateUser` — создание пользователя с проверкой уникальности логина.
    - `GetUserById` — получение пользователя по id.
    - `GetUserByName` — получение пользователя по имени и фамилии.
    - `UpdateUser` — обновление данных пользователя (кроме id и логина).
    - `DeleteUser` — удаление пользователя по id.
3. Использование PostgreSQL для хранения данных.
4. Реализация функции/процедуры для каждого метода, а также создание таблиц и индексов в `.sql` файлах.
5. Валидации с использованием библиотеки FluentValidation и маппинг с помощью Mapperly.
6. Использование Dapper для маппинга и выполнения запросов.

# Лабораторная работа 2: Writer (Rate Limiting)

**Задача**: Написать сервис для выполнения CRUD операций с ограничениями на количество запросов в минуту (Rate limiting) для методов UserService.

## Шаги:
1. Создание модели `RateLimit` с полями `route`, `requests_per_minute`.
2. Реализация методов:
    - `CreateLimit` — создание ограничения с проверкой уникальности по роуту.
    - `GetLimitByRoute` — получение ограничения по роуту.
    - `UpdateLimit` — обновление ограничения.
    - `DeleteLimit` — удаление ограничения.
3. Использование MongoDB для хранения данных о лимитах.
4. Валидации с использованием FluentValidation.

# Лабораторная работа 3: Reader (Подписка на изменения)

**Задача**: Написать первую часть сервиса, который будет загружать ограничения из MongoDB в память и отслеживать их изменения с использованием подписок на изменения.

## Шаги:
1. Загрузка ограничений из коллекции `rate_limits` в память с использованием `FindOptions` и `BatchSize`.
2. Обработка событий с использованием `collection.WatchAsync()` для отслеживания изменений в коллекции.
3. Реализация метода `GetAllLimits` для получения всех загруженных ограничений из памяти.
4. Обработка обновлений в отдельном потоке для поддержания актуальности данных.

# Лабораторная работа 5: Reader (Обработка запросов с Redis)

**Задача**: Реализовать вторую часть функциональности сервиса Reader, отслеживающего превышение лимита RPM для методов UserService с использованием Kafka и Redis.

## Шаги:
1. Подключение к Kafka для получения запросов пользователей.
2. Отслеживание превышения лимита RPM для каждого пользователя и установка флага в Redis.
3. Управление TTL флага в Redis (5 минутный запрет на выполнение запросов).

**Задача**: Реализовать Interceptor, который проверяет наличие `user_id` в заголовках запросов и блокирует запросы без него.
